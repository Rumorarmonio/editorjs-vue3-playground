# Техническое задание

## Проект: визуальный редактор контента на Vue 3 + TypeScript + Vite + Editor.js

---

## 1. Назначение документа

Данный документ фиксирует требования к новому проекту визуального редактора контента на базе **Vue 3 Composition API**, **TypeScript**, **Vite** и **Editor.js**.

Документ предназначен для:

- ручной реализации проекта;
- реализации проекта с помощью AI-агентов;
- контроля порядка работ;
- фиксации архитектурных решений;
- снижения риска хаотичного роста сложности.

Документ описывает:

- цель и scope проекта;
- желательную архитектуру;
- требования к структуре директорий и файлов;
- требования к SCSS-архитектуре;
- требования к editor-layer и renderer-layer;
- обязательный порядок реализации;
- состав стандартных и кастомных возможностей;
- критерии готовности этапов;
- спорные архитектурные вопросы и допустимые варианты решений.

Документ должен использоваться как **основная техническая спецификация** проекта.

---

## 2. Общая идея проекта

Текущий проект является **промежуточным standalone SPA-проектом**, который должен:

- позволить отработать архитектуру Editor.js-проекта на Vue 3;
- позволить безопасно реализовать и проверить block tools, inline tools, tunes, nested editors, renderer и JSON workflow;
- быть пригодным для деплоя на **GitHub Pages**;
- выступать подготовительным этапом перед следующим, более полноценным демо-проектом на **Nest + Nuxt**, где появятся:
    - настоящая админка;
    - создание страниц как сущностей;
    - backend persistence;
    - SSR на стороне Nuxt;
    - page-level settings и layout-level логика.

Текущий проект **не должен превращаться в полноценную CMS**. Его задача — построить качественную и расширяемую фронтенд-основу редактора и renderer-а.

---

## 3. Цель проекта

Необходимо реализовать фронтенд-проект визуального редактора контента, в котором:

1. Контент редактируется в **Editor.js**.
2. Результат сохраняется в JSON-формате.
3. JSON может быть просмотрен на отдельной preview/render-странице.
4. На ранних этапах проект работает **без бэкенда**.
5. Архитектура позволяет в дальнейшем без болезненной перестройки добавлять:
    - кастомные block tools;
    - кастомные inline tools;
    - block tunes;
    - nested editors;
    - сложные composite blocks;
    - позднюю валидацию;
    - upload-интеграции;
    - административную обвязку.

Главный принцип проекта:

> Сначала реализуется стабильная и хорошо проверенная база на стандартных инструментах Editor.js. Только после этого добавляются расширения, кастомные блоки, tunes,
> nested editors и поздние улучшения.

## 4. Scope проекта

### 4.1. Что входит в scope

В scope проекта входят:

- права доступа;

- полноценный backend API;
- production-ready upload backend;
- SSR текущего проекта;
- полноценная page-management админка.

### 4.3. Что относится к поздним, но допустимым расширениям

Допускается реализация на поздних этапах:

- `zod`-валидации;
- `imask` и input masks;
- ручного `Import JSON`;
- продвинутых tunes;
- Fancybox deep-linking;
- sidebar-навигации по блокам;
- background/spoiler tunes;
- расширенной media-логики;
- collapse/expand поведения у вложенных пунктов sidebar navigation.

- рабочий Editor.js внутри Vue-приложения;
- подключение стандартных block tools;

## 7. Общие архитектурные принципы

### 7.1. Архитектура должна быть заложена с самого начала

С самого начала проект должен иметь корректное разделение слоёв, но без преждевременного усложнения.

Это означает:

- правильное разделение `shared / admin / renderer` необходимо сразу;
- тяжёлые абстракции не должны внедряться заранее без подтверждённой необходимости;
- архитектура должна расширяться по мере появления реальной сложности.

### 7.2. Базовый приоритет реализации

Приоритет реализации должен быть следующим:

1. Архитектурный каркас.
2. Shared-слой и registry блоков.
3. Источник данных и базовый editor shell.
4. Стандартные block tools.
5. Стандартные inline tools.
6. Renderer.
7. Export / Reset draft UX.
8. Базовые tunes.
9. Plain field system.
10. Первый простой custom block.
11. Первый nested editor scenario.
12. Composite blocks.
13. Обязательные rich fields.
14. Кастомные inline tools.
15. Обязательный media gallery / slider block.
16. Sidebar navigation из JSON.
17. Поздние utility-сценарии (`Import JSON`).
18. Валидация.
19. Маски.
20. Некритичные улучшения.

### 7.3. Принцип кастомизации

Кастомизация должна начинаться только после того, как:

- понятна базовая JSON-модель;
- стабильно работает `save/load`;
- проверена совместимость стандартных block tools и inline tools;
- готов renderer для стандартных блоков.

### 7.4. Inline tools как общий принцип

Все доступные inline tools — стандартные и кастомные — должны быть включены в **максимально возможном количестве совместимых block tools**.

Это означает:

- если конкретный block tool или nested editor технически поддерживает `inlineToolbar`, он должен быть включён;
- исключения должны быть явными, обоснованными и зафиксированными;
- inline tools должны быть доступны в paragraph, header, list и во всех rich field scenarios, где это имеет смысл.

## 8. Требования к структуре проекта

Рекомендуемая структура проекта:

```text
src/
  content/
    default-page.json

  editor/
    shared/
      types/
      entities/
      blocks/
      registry/
      constants/
      navigation/

    admin/
      core/
      config/
      tools/
        blocks/
        inline/
      tunes/
      fields/
      helpers/
      styles/

    renderer/
      components/
        blocks/
        navigation/
      helpers/
      types/

  pages/
    EditorPage/
    PreviewPage/

  components/
  composables/
  utils/
  styles/
```

### 8.1. Смысловое разделение слоёв

#### `editor/shared`

Содержит всё, что используется и в editor-layer, и в renderer-layer:

- типы данных;
- registry блоков;
- block data types;
- entity types;
- JSON-модели;
- общие константы;
- helper-типизацию для navigation tree.

#### `editor/admin`

Содержит всё, что относится к редактированию:

- создание editor config;
- block tools;
- inline tools;
- tunes;
- field builders;
- nested editor helpers/managers;
- editor infrastructure.

#### `editor/renderer`

Содержит всё, что относится к рендеру JSON:

- Vue-компоненты блоков;
- mapping `block.type -> component`;
- renderer utilities;
- renderer types;
- sidebar/navigation rendering.

### 8.2. Правило структуры компонентов

Если компоненту требуются модульные стили, предпочтительная структура должна быть следующей:

```text
ComponentName/
  ComponentName.vue
  ComponentName.module.scss
```

Это правило особенно рекомендуется для:

- renderer-блоков;
- editor-shell компонентов;
- сложных UI-компонентов;
- блоков с собственной разметкой и логикой.

Для очень простых компонентов без собственных стилей отдельная директория не является обязательной, но желательна единообразная структура.

## 9. Требования к SCSS-архитектуре

### 9.1. Основной подход

Все Vue-компоненты должны в первую очередь стилизоваться через **модульные SCSS-файлы**.

Формат подключения стилей:

```vue

<style module lang="scss" src="./TwoColumns.module.scss" />
```

Этот подход обязателен для:

- обычных Vue-компонентов;
- renderer-компонентов блоков;
- страниц;
- editor-shell компонентов, если они реализованы как Vue-компоненты.

### 9.2. Подключение глобальных SCSS-зависимостей

Необходимо использовать глобальное подключение SCSS entrypoint через `vite.config.ts`, чтобы:

- переменные;
- функции;
- миксины;
- media helpers;
- общие SCSS-модули

не импортировались вручную в каждом `.module.scss`-файле.

Следует исходить из сценария, в котором глобальный SCSS entrypoint подмешивается через `additionalData`.

### 9.3. Базовая структура SCSS

За основу следует взять полноценную структуру SCSS-слоя с разделением на:

- `styles/utils/`
- `styles/utils/typography/`
- `styles/components/`
- `styles/_main.scss`
- `styles/_global.scss`
- `styles/common.scss`
- `styles/css-variables.scss`
- `styles/utilities.scss`

### 9.4. Глобальный SCSS-слой

Глобальный SCSS-слой должен содержать:

- переменные;
- функции;
- миксины;
- media helpers;
- CSS variables;
- utilities;
- reset/common rules.

### 9.5. Модульный SCSS-слой

Модульный SCSS-слой должен использоваться для стилизации конкретных Vue-компонентов.

### 9.6. Дополнительный global editor-specific слой

Допускается наличие дополнительных файлов наподобие:

- `editor-renderer.scss`
- `editor-admin.scss`

как **резервного вспомогательного слоя**, если для некоторых editor-related сущностей предпочтительнее использовать глобальные классы вместо модульных.

Важно:

- этот слой не является основным способом стилизации renderer-компонентов;
- renderer-компоненты всё равно должны в первую очередь использовать module SCSS;
- editor-specific global files допустимы как инфраструктурный слой и как запасной вариант.

### 9.7. Анимации и интеракции

Необходимо принять единое правило: все базовые интеракции и анимации должны иметь длительность **0.3s**.

Это относится к:

- hover-состояниям;
- toggle-переключателям;
- открытию/закрытию select;
- подсветке фокуса;
- появлению ошибок;
- служебным editor UI-переходам.

Следует использовать единый SCSS-токен перехода.

## 10. Требования к модели данных

### 10.1. Registry-подход

В проекте должен использоваться единый registry блоков.

Registry должен быть единым источником истины для:

- editor-layer;
- renderer-layer;
- типизации блока по `type`;
- типизации `data` конкретного блока;
- типизации renderer props.

### 10.2. Discriminated union

Все блоки должны быть представлены как discriminated union по `type`.

Следует использовать общий тип блока, выводимый из registry.

### 10.3. Разделение типов

Не допускается сваливание всех типов в один большой файл.

Типы должны быть разделены минимум на:

- editor output types;
- registry types;
- block data types;
- entity types;
- navigation types.

### 10.4. JSON-модель

В качестве основной модели editor output необходимо использовать JSON-структуру, совместимую с `OutputData` Editor.js.

Допустимы alias-типы, но не следует создавать вторую конкурирующую модель без необходимости.

## 11. Источник данных и хранение

### 11.1. Источник данных по умолчанию

Необходимо использовать статический JSON-файл проекта:

- `src/content/default-page.json`

### 11.2. Draft-хранение

Во время редактирования актуальный черновик должен сохраняться в `localStorage`.

### 11.3. Приоритет источников данных

При открытии редактора и preview необходимо использовать следующий приоритет:

1. Draft из `localStorage`, если он существует.
2. `default-page.json`, если draft отсутствует.

### 11.4. Обязательные действия первой версии

Для первой версии обязательно реализовать:

- `Export JSON`
- `Reset draft`
- disabled state для `Export JSON`

### 11.5. Поведение Export JSON

`Export JSON` должен:

- брать актуальные данные редактора;
- экспортировать их в `.json` файл;
- быть disabled, если отсутствует локальный draft или отсутствуют изменения относительно исходного состояния.

### 11.6. Поведение Reset draft

`Reset draft` должен:

- удалять локальный draft из `localStorage`;
- сбрасывать editor state;
- возвращать редактор и preview к содержимому `default-page.json`.

### 11.7. Import JSON

`Import JSON` не относится к обязательным функциям первой версии.

Он может быть добавлен как поздний utility-сценарий и, если реализуется, должен:

- принимать `.json` файл от пользователя;
- проверять корректность файла как минимум на базовом уровне;
- загружать содержимое в editor state;
- записывать его в `localStorage` как текущий draft.

### 11.8. Базовая валидация Import JSON

Если `Import JSON` реализуется, на первом шаге необходима только базовая валидация:

- файл читается как JSON;
- верхний уровень является объектом;
- у объекта есть `blocks`;
- `blocks` — массив.

Глубокая проверка структуры всех блоков относится к поздним этапам.

### 11.9. Браузерная запись в проектный JSON

Браузер не должен пытаться напрямую обновлять JSON-файл проекта как обычный файл исходного кода.

Обновление проектного JSON должно происходить вручную: через экспорт и дальнейшее сохранение в репозиторий.

## 12. Требования к editor-layer

### 12.1. Editor config

Необходимо реализовать фабрику конфигурации редактора, например `createEditorConfig`, которая:

- принимает стартовые данные;
- подключает block tools;
- подключает inline tools;
- подключает tunes;
- задаёт `onChange`;
- возвращает единый конфиг Editor.js.

### 12.2. Базовый editor-shell

Необходимо реализовать editor-shell на Vue, который:

- создаёт и уничтожает Editor.js instance;
- получает initial data;
- обрабатывает save/update;
- показывает служебные действия редактора.

### 12.3. Прямое использование `window.EditorJS`

Необходимо избегать зависимости от глобального `window.EditorJS`, если используется модульная установка через npm.

### 12.4. Инициализация без `setTimeout`

Необходимо стремиться к предсказуемому lifecycle.

Если вложенные editor instances потребуются позже, их инициализация должна строиться на явном lifecycle-менеджменте, а не на случайных `setTimeout`.

## 13. Стандартные block tools: обязательный первый функциональный этап

### 13.1. Цель этапа

Поднять рабочий Editor.js на стандартных блоках и получить корректный JSON output.

### 13.2. Базовый набор block tools

На старте должны быть подключены:

- Paragraph
- Header
- List
- Image
- Quote
- Delimiter
- Table
- Embed

### 13.3. Требования к List tool

В проекте не должно планироваться два независимых пользовательских плагина — “одномерный список” и “многомерный список”.

Следует использовать **один основной list tool**, который:

- на первом этапе работает как ordered/unordered list;
- на следующем этапе расширяется до nested list;
- не дублируется вторым отдельным list-плагином без крайней необходимости.

### 13.4. Что необходимо проверить

После реализации этапа необходимо проверить:

- редактор успешно инициализируется;
- все подключённые block tools отображаются;
- блоки можно добавлять, редактировать, удалять и перемещать;
- `save()` возвращает валидный JSON;
- сохранённый JSON можно повторно загрузить в editor;
- типы `type` в output совпадают с ключами tools конфигурации.

### 13.5. Критерии готовности этапа

Этап считается завершённым, если:

- редактор стабильно работает на стандартных блоках;
- save/load проходят без потери данных;
- нет необходимости в кастомных block tools для запуска базовой версии.

---

## 14. Стандартные inline tools: обязательный второй функциональный этап

### 14.1. Цель этапа

Подключить и проверить стандартные inline tools до начала кастомизации.

### 14.2. Базовый набор inline tools

На старте должны быть подключены:

- bold
- italic
- link
- underline
- marker (для фона текста)
- strikethrough

`inline code` допускается добавить позже как необязательный инструмент.

### 14.3. Требование к strikethrough

`Strikethrough` должен считаться обязательным inline tool проекта.

Предпочтительный порядок реализации:

1. Сначала попытаться использовать готовый сторонний совместимый plugin.
2. Если сторонний plugin не проходит проверку стабильности и совместимости, реализовать собственный strikethrough tool.

### 14.4. Доступность inline tools

Все стандартные inline tools должны быть включены в максимально возможном количестве совместимых блоков, включая:

- paragraph;
- header;
- list;
- rich field scenarios на nested Editor.js.

### 14.5. Проверка совместимости

После реализации необходимо проверить комбинации:

- bold + italic
- bold + link
- underline + link
- marker + link
- marker + underline
- strikethrough + bold
- strikethrough + italic
- strikethrough + link
- strikethrough + underline
- strikethrough + marker
- bold + italic + link
- marker + bold + link

Также необходимо проверить:

- повторное открытие ранее сохранённого контента;
- корректность рендера комбинаций;
- отсутствие потери разметки после повторного `save()`.

### 14.6. Inline tools в списках

Для list tool и nested list scenarios необходимо явно включить `inlineToolbar`, если требуется поддержка inline formatting в пунктах списка.

### 14.7. Критерии готовности этапа

Этап считается завершённым, если:

- стандартные inline tools работают стабильно;
- базовые комбинации инструментов не ломаются;
- сохранение и повторная загрузка не искажают контент;
- inline tools доступны в максимально возможном количестве совместимых tool-сценариев.

---

## 15. Renderer: обязательный ранний этап

### 15.1. Цель этапа

Реализовать отдельный renderer JSON-контента на Vue-компонентах.

### 15.2. Требования к renderer

Renderer должен:

- принимать JSON editor output;
- по `block.type` определять нужный Vue-компонент;
- передавать в компонент типизированные `data`;
- корректно рендерить стандартные block tools.

### 15.3. Стили renderer-компонентов

Renderer-компоненты должны в первую очередь использовать **module SCSS**.

Отдельный глобальный `editor-renderer.scss` допускается оставить как резервный слой, но не как основной способ стилизации renderer.

### 15.4. Проверка

Необходимо проверить:

- корректный рендер всех стандартных block tools;
- совпадение структуры JSON и renderer-модели;
- отсутствие зависимости renderer от editor runtime.

### 15.5. Критерии готовности этапа

Этап считается завершённым, если:

- preview-страница корректно рендерит JSON;
- renderer отделён от editor-layer;
- компоненты renderer используют типизированные данные.

## 16. Block Tunes: первый этап кастомизации

### 16.1. Общий подход

Block Tunes должны использоваться для дополнительных метаданных блока, а не для внутренней логики конкретного block tool.

### 16.2. Приоритет tunes

Порядок реализации:

1. `AnchorTune`
2. `SpacingTune`
3. `LabelTune` / `TitleTune`
4. `BackgroundTune` — поздний этап
5. `SpoilerTune` — поздний этап

### 16.3. AnchorTune

Должен хранить значение якоря блока и использоваться для скролла по ссылкам вида `#anchor`.

### 16.4. SpacingTune

Должен быть основным решением для задания верхних и/или нижних отступов блока.

### 16.5. LabelTune / TitleTune

Должен хранить label для навигации по странице и использоваться для sidebar-навигации.

### 16.6. BackgroundTune

Допускается как поздний tune для выбора фонового режима блока.

### 16.7. SpoilerTune

Допускается как поздний tune для скрываемых/раскрываемых блоков.

### 16.8. Проверка

После реализации каждого tune необходимо проверить:

- сохранение состояния tune;
- повторную загрузку состояния;
- применение данных tune в renderer;
- отсутствие влияния на блоки без tune-data.

### 16.9. Критерии готовности этапа

Tune считается реализованным, если:

- значение можно изменить в editor UI;
- значение сохраняется в JSON;
- renderer корректно использует значение;
- блок без заданного tune продолжает работать штатно.

## 17. Поля в custom blocks

### 17.1. Общий подход

В проекте должны поддерживаться два типа полей:

- `plain fields`
- `rich fields`

### 17.2. Plain fields

К plain fields относятся:

- text input
- textarea
- select
- radio group
- toggle
- image/url/file selectors

На ранних этапах необходимо реализовать именно plain fields.

### 17.3. Toggle

Checkbox в editor UI должен иметь внешний вид toggle-переключателя, при этом на уровне данных он остаётся обычным boolean.

### 17.4. Rich fields

Rich fields должны быть предусмотрены архитектурно **с самого начала** и **обязательно реализованы позднее**.

Rich fields не являются опциональной возможностью “только если понадобится”.

К rich fields относятся:

- nested paragraph field;
- nested header field;
- другие single-purpose nested editors.

### 17.5. Архитектурное требование

Одно и то же логическое поле должно допускать в будущем две реализации:

- как plain field;
- как rich field.

### 17.6. Ограничение plain fields

Обычные `input` и `textarea` не должны рассматриваться как носители inline tools.

Если полю нужны inline tools, оно должно быть реализовано как rich field на nested Editor.js.

### 17.7. Field wrappers

Для plain field system необходимо использовать единообразную структуру поля, включающую:

- field wrapper;
- label;
- control;
- optional hint;
- optional error.

Под field wrapper понимается обычная UI-обёртка поля, а не отдельная технология.

### 17.8. Медиа-поля в custom blocks

Для кастомных блоков загрузка и выбор изображений/видео должны реализовываться прежде всего через **отдельные media fields**, а не через nested Editor.js с
единственным image tool.

Nested Editor.js для image/video не должен считаться базовым способом работы с media в custom blocks.

## 18. Первый custom block: только после tunes и plain fields

### 18.1. Цель этапа

Реализовать первый простой custom block без nested editors.

### 18.2. Обязательные тренировочные custom blocks

Первые простые custom blocks должны выбираться из следующего набора:

- CTA
- Section Intro
- Notice
- Banner
- SpacerBlock (только как дополнительный блок, если действительно понадобится)

### 18.3. Роли и примерные данные блоков

#### CTA

Назначение: блок призыва к действию.

Примерные поля:

- title
- description
- buttonText
- buttonUrl
- theme / variant
- optional image

#### Section Intro

Назначение: вводный блок секции.

Примерные поля:

- title
- subtitle / short text
- optional aside
- optional meta/tune data

#### Notice

Назначение: информационный или предупреждающий блок.

Примерные поля:

- title
- text
- type (`info | warning | success | error`)

#### Banner

Назначение: баннерный/hero-like блок.

Примерные поля:

- title
- description
- image
- buttonText
- buttonUrl
- theme
- mediaPosition

#### SpacerBlock

Назначение: самостоятельный layout-gap блок.

Примерные поля:

- spacing preset

### 18.4. Требования к custom block

Каждый custom block должен иметь:

- собственный data type;
- renderer component;
- tool class;
- save/load сценарий;
- при необходимости — tune support;
- при необходимости — простые поля.

### 18.5. Что проверить

- блок можно создать;
- блок можно удалить;
- блок корректно сохраняет данные;
- блок корректно восстанавливается из JSON;
- renderer умеет его отрисовывать.

### 18.6. Критерии готовности этапа

Этап считается завершённым, если как минимум один custom block полностью проходит цикл `create -> edit -> save -> reload -> render`.

## 19. Nested editors: отдельный этап

### 19.1. Общий принцип

Nested editors должны добавляться только после того, как:

- работают plain fields;
- реализованы tunes;
- есть хотя бы один простой custom block;
- отлажен базовый lifecycle editor.

### 19.2. Первый nested editor

Первым должен быть реализован **один простой custom block с одним nested editor**.

Не следует начинать сразу с `TwoColumns`.

### 19.3. Ограничение nested editors

Во вложенном editor instance должен быть жёстко ограничен набор tools.

Примеры:

- только `paragraph`;
- только `header`;
- `paragraph + list`.

### 19.4. Single-purpose nested editors

Необходимо предусмотреть режим nested editor, который используется как почти “одно поле”, например:

- rich paragraph field;
- rich header field.

### 19.5. Header-field стратегия

`Header` tool нативно поддерживает выбор уровня заголовка.

Следовательно, долгосрочное целевое решение для rich header field — вложенный editor на основе official `Header` tool, а не только комбинация `input + select`.

На раннем этапе допускается временная реализация `input + select(level)`, но это не должно считаться финальным целевым решением.

### 19.6. Ограничение одного блока

Если вложенный editor используется как одно rich-поле, необходимо своей логикой ограничивать или валидировать сценарий, в котором допускается только один block.

### 19.7. Проверка

- nested editor корректно создаётся;
- nested editor корректно уничтожается;
- save/load nested data работают стабильно;
- ограниченный whitelist tools соблюдается;
- renderer корректно рендерит вложенный контент.

### 19.8. Критерии готовности этапа

Этап считается завершённым, если существует хотя бы один custom block с корректно работающим single-purpose nested editor.

## 20. Rich fields: обязательный поздний этап

### 20.1. Общий принцип

После plain field system и первого nested editor scenario необходимо обязательно реализовать reusable rich fields.

### 20.2. Обязательные rich field scenarios

Необходимо реализовать как минимум:

- `RichParagraphField`
- `RichHeaderField`

### 20.3. Требования к rich fields

Rich fields должны:

- использовать nested Editor.js;
- иметь ограниченный whitelist tools;
- поддерживать inline tools;
- корректно сохраняться и восстанавливаться;
- рендериться как обычные типизированные поля блока.

### 20.4. Что проверить

- rich field создаётся корректно;
- rich field стабильно работает как nested editor;
- inline tools доступны;
- save/load не искажают данные;
- renderer корректно рендерит содержимое.

### 20.5. Критерии готовности этапа

Этап считается завершённым, если reusable rich fields реализованы и могут использоваться внутри последующих custom blocks.

## 21. TwoColumns и другие composite blocks

### 21.1. Порядок

`TwoColumns` и другие сложные composite blocks должны реализовываться только после успешного этапа с одним nested editor и базовыми rich fields.

### 21.2. Требования к TwoColumns

`TwoColumns` должен хранить:

- layout variant;
- direction / reversed state;
- tune data;
- left content;
- right content.

`left` и `right` должны быть представлены как отдельные nested editor outputs.

### 21.3. Разрешённые инструменты во вложенных колонках

Для колонок должен использоваться ограниченный whitelist tools, например:

- paragraph
- header
- list

### 21.4. Проверка

- корректный save/load обеих колонок;
- отсутствие конфликтов toolbar;
- корректный renderer;
- стабильное поведение при удалении и перемещении блока.

## 22. Кастомные inline tools

### 22.1. Порядок реализации

Кастомные inline tools должны реализовываться только после стандартных inline tools и после появления стабильных nested editor scenarios.

### 22.2. Обязательные inline tools

Следует реализовать:

1. Стандартный `Marker` для цвета фона текста.
2. Кастомный inline tool для цвета текста.

### 22.3. Общий подход к кастомному inline tool

Кастомный tool для цвета текста должен:

- использовать собственную inline-обёртку;
- использовать собственный CSS class;
- не мутировать существующие `b`, `i`, `a`, `u` и другие inline-элементы;
- корректно сохраняться и восстанавливаться.

### 22.4. Доступность custom inline tools

Кастомный inline tool для цвета текста должен включаться в максимально возможном количестве совместимых block tools и rich field scenarios.

### 22.5. Проверка

Необходимо проверить комбинации:

- text color + bold
- text color + italic
- text color + link
- text color + underline
- text color + background marker
- text color + strikethrough
- text color + несколько стандартных inline tools одновременно

### 22.6. Критерии готовности этапа

Этап считается завершённым, если кастомный inline tool:

- работает в nested editor и/или основном editor;
- корректно сохраняется;
- корректно рендерится;
- не ломает комбинации со стандартными inline tools.

## 23. Media gallery / slider block: обязательный composite-блок

### 23.1. Общий принцип

Media gallery / slider block является обязательным кастомным блоком и должен быть реализован поэтапно.

### 23.2. Обязательное технологическое требование

Для всех слайдерных сценариев необходимо использовать **Swiper**.

Самописный слайдер не допускается как базовое решение.

### 23.3. Рекомендуемая стратегия

На первом этапе следует рассматривать **один общий media block** с режимом:

- `slider`
- `gallery`

Если позже различие по UX и структуре данных окажется слишком большим, допускается разделение на два независимых блока.

### 23.4. Этап 1 media-block

На первом этапе у каждого слайда должен быть:

- один media item (`image | video`);
- caption;
- optional rich description.

У блока должны быть настройки:

- `mode: slider | gallery`
- `galleryId`
- `enableFancybox`
- `syncUrlWithFancybox`

### 23.5. Этап 2 media-block

На втором этапе допускается расширение, при котором у одного слайда:

- несколько media items;
- одно общее description;
- более сложная внутренняя media-структура.

### 23.6. Fancybox grouping и URL

Необходимо предусмотреть:

- группировку media items по `galleryId`;
- возможность открытия конкретного элемента по URL;
- смену URL при перелистывании в Fancybox;
- восстановление состояния после перезагрузки страницы, если URL указывает на конкретный media item.

На раннем этапе допускается реализация через индекс, но долгосрочно предпочтительнее стабильный slide/media id.

### 23.7. Caption и description

Необходимо предусмотреть:

- caption для показа в карточке и/или Fancybox;
- optional rich description через nested Editor.js.

### 23.8. Тогглы и дополнительные поля блока

В блоке должны быть предусмотрены:

- toggle для `enableFancybox`;
- toggle для `syncUrlWithFancybox`;
- plain input для `galleryId` / viewer id.

### 23.9. Проверка

Необходимо проверить:

- render как `slider`;
- render как `gallery`;
- корректную работу Swiper;
- корректную работу Fancybox;
- корректную группировку по `galleryId`;
- корректную работу caption;
- корректный save/load JSON.

### 23.10. Критерии готовности этапа

Этап считается завершённым, если реализован минимум первый этап media gallery/slider block со стабильным save/load/render и viewer behavior.

## 24. List tool и nested list: обязательный поэтапный сценарий

### 24.1. Общий принцип

В проекте не требуется отдельный самостоятельный “одномерный list plugin”.

Нужен **один основной list tool**, который проходит два этапа развития.

### 24.2. Этап 1

List tool должен поддерживать:

- ordered list;
- unordered list;
- inline tools в пунктах списка.

На этом этапе список может использоваться как обычный одноуровневый список.

### 24.3. Этап 2

Тот же list tool должен быть доведён до nested list behavior:

- вложенные уровни;
- ordered/unordered на всех уровнях;
- проверка вложенности и UX переходов.

### 24.4. Поздняя исследовательская задача

После всех основных этапов допускается исследование сценария, в котором тип списка (`ordered/unordered`) задаётся отдельно для каждого уровня.

Это не относится к базовой обязательной реализации.

### 24.5. Проверка

Необходимо проверить:

- корректный save/load list data;
- корректный рендер списков;
- inline tools в элементах списка;
- переход от простого режима к nested режиму без смены пользовательского блока.

## 25. Sidebar navigation по контенту: поздний этап

### 25.1. Общий принцип

Sidebar navigation должна строиться **из JSON editor content**, а не путём сканирования итогового DOM после рендера страницы.

JSON является источником истины для построения navigation-структуры.

### 25.2. Базовые источники данных navigation

На позднем этапе sidebar navigation должна использовать:

- `AnchorTune`
- `LabelTune`
- при необходимости уровень заголовка `Header.level`

### 25.3. Этап 1: одномерная navigation

На первом этапе необходимо реализовать:

- построение плоского списка пунктов меню из JSON;
- использование `label + anchor`;
- рендер sidebar как обычного списка ссылок.

### 25.4. Этап 2: вложенная navigation

На втором этапе необходимо реализовать:

- определение вложенности пунктов;
- построение navigation tree;
- использование уровней заголовков как основного сигнала вложенности.

### 25.5. Позднее поведение

После реализации tree-navigation допускается добавить:

- collapse / expand;
- accordion behavior;
- дополнительные опции отображения.

### 25.6. Что не требуется на текущем этапе

Sidebar navigation не должна зависеть от будущих page-level settings следующего fullstack-проекта.

Однако архитектурно её следует строить так, чтобы позднее можно было передавать внешние `sidebar options` на уровне страницы.

### 25.7. Проверка

Необходимо проверить:

- корректное извлечение navigation data из JSON;
- корректную генерацию flat navigation;
- корректную генерацию tree navigation на втором этапе;
- корректный рендер sidebar;
- корректный скролл к anchor.

## 26. Архитектура custom tools и helpers

### 26.1. Общий принцип

Необходимо избегать монолитных helper-классов наподобие универсального `EditorFields`, который знает сразу обо всех типах полей, upload, nested editors и логике tool
lifecycle.

### 26.2. Предпочтительная стратегия

Следует использовать набор маленьких специализированных модулей:

- builders для plain fields;
- helpers для image/media fields;
- helpers для nested editor shell;
- managers для lifecycle nested editors;
- отдельные helpers для selection/inline logic;
- отдельные helpers для DOM building.

### 26.3. Наследование и композиция

По умолчанию должна использоваться **композиция**, а не глубокая иерархия наследования.

Узкие base classes допустимы позже, если появится реальная повторяемость между custom block tools, inline tools или tunes.

### 26.4. Передача контекста

Не следует строить архитектуру на передаче целого `this` во все helper-классы.

Следует передавать только необходимые зависимости явно.

## 28. Валидация и masks: поздний этап

### 28.1. Приоритет

Валидация не относится к ранним этапам и должна появляться ближе к завершению базовой редакторной функциональности.

### 28.2. Библиотеки

Допускается использовать:

- `zod` — для схем и декларативной валидации;
- `imask` — для input masks.

### 28.3. Уровни валидации

#### Plain fields

Ошибки должны отображаться под конкретным полем.

#### Nested editors

Ошибки должны отображаться под конкретным nested editor как под цельным полем.

#### Внутренние блоки nested editor

Отдельная адресная валидация внутренних paragraph/header/list block-элементов не является обязательной для первой версии и может быть отложена.

### 28.4. Минимальная валидация

На первом шаге валидации достаточно:

- required;
- min/max length;
- email;
- url;
- phone pattern.

### 28.5. Полная валидация

Более сложная валидация и маски относятся к одному из последних этапов.

### 28.6. Критерии готовности этапа

Этап считается завершённым, если:

- plain fields поддерживают базовые ошибки;
- nested editor fields поддерживают ошибки на уровне поля;
- ошибки отображаются предсказуемо;
- сохранение и экспорт можно ограничить при критичных ошибках.

## 30. Спорные и неоднозначные архитектурные вопросы

### 30.1. SpacerBlock vs SpacingTune

#### Предпочтительное решение

Основным механизмом отступов должен быть `SpacingTune`.

#### Допустимая альтернатива

`SpacerBlock` может быть добавлен позже как отдельный технический блок, если редактору действительно понадобится самостоятельный пустой layout-блок.

### 30.2. Plain field vs rich field

#### Предпочтительное решение

По умолчанию следует использовать plain fields.

#### Обязательное расширение

После plain field system необходимо обязательно реализовать rich fields.

### 30.3. Заголовок как input или как nested header editor

#### Предпочтительное долгосрочное решение

Nested header editor с inline tools и нативным выбором уровня заголовка.

#### Допустимое раннее решение

`input + select(level)` как временный первый этап.

### 30.4. Marker vs custom text color tool

#### Зафиксированное решение

Для фона текста используется стандартный `Marker`.

Для цвета текста реализуется отдельный кастомный inline tool.

### 30.5. Class vs data-attributes для кастомных inline стилей

#### Предпочтительное решение

Использовать CSS classes.

#### Допустимая альтернатива

`data-*` может быть выбран позднее, если появится необходимость в более абстрактной theme-model.

### 30.6. Степень раннего архитектурного усложнения

#### Предпочтительное решение

Лёгкий каркас с постепенным ростом архитектуры.

#### Что не рекомендуется

Полноценный heavy framework для tools/tunes/helpers до появления реальной сложности.

### 30.7. Media block: один общий блок или два

#### Предпочтительное решение

Начинать с одного общего media block с режимом `slider | gallery`.

#### Допустимая альтернатива

Если позже различие по UX и данным станет слишком сильным, разделить на два отдельных блока.

### 30.8. Sidebar navigation: flat или tree

#### Предпочтительное решение

Сначала реализовать flat navigation.

#### Позднее расширение

Позже реализовать tree navigation, строящуюся в первую очередь по уровням заголовков.
