# Техническое задание

## Проект: визуальный редактор контента на Nuxt + TypeScript + Editor.js

---

## 1. Назначение документа

Данный документ фиксирует требования к новому проекту визуального редактора контента на базе **Nuxt**, **Vue 3 Composition API**, **TypeScript** и **Editor.js**.

Документ предназначен для:

- ручной реализации проекта;

- синхронизации ожидаемого поведения editor-layer, renderer-layer и вспомогательной инфраструктуры.

Документ описывает:

## 2. Общая идея проекта

Текущий проект является **промежуточным standalone static/prerender Nuxt-проектом**, который должен:

- позволить отработать архитектуру Editor.js-проекта на Nuxt;
- позволить безопасно реализовать и проверить block tools, inline tools, tunes, nested editors, renderer и JSON workflow;
- быть пригодным для деплоя на **GitHub Pages**;
- выступать подготовительным этапом перед следующим, более полноценным демо-проектом на **Nest + Nuxt**, где появятся:
    - настоящая админка;
    - создание страниц как сущностей;
    - backend persistence;
    - SSR на стороне Nuxt;
    - page-level settings и layout-level логика;
    - полноценное управление страницами и их настройками.

Текущий проект **не должен превращаться в полноценную CMS**. Его задача — построить качественную и расширяемую фронтенд-основу редактора, renderer-а и связанных с
ними UI/UX-сценариев.

---

## 3. Цель проекта

Необходимо реализовать фронтенд-проект визуального редактора контента, в котором:

1. Контент редактируется в **Editor.js**.
2. Результат сохраняется в JSON-формате.
3. JSON может быть просмотрен на отдельной preview/render-странице.
4. На ранних этапах проект работает **без бэкенда**.
5. Архитектура позволяет в дальнейшем без болезненной перестройки добавлять:
    - кастомные block tools;
    - кастомные inline tools;
    - block tunes;
    - nested editors;
    - сложные composite blocks;
    - позднюю валидацию;
    - upload-интеграции;
    - административную обвязку;
    - позднюю локализацию UI редактора;
    - позднюю поддержку светлой/тёмной темы;
    - поздние accessibility-улучшения, включая клавиатурную навигацию.

Главный принцип проекта:

> Сначала реализуется стабильная и хорошо проверенная база на стандартных инструментах Editor.js. Только после этого добавляются расширения, кастомные блоки, tunes,
> nested editors и поздние улучшения.

## 4. Scope проекта

### 4.1. Что входит в scope

В scope проекта входят:

- Editor.js в составе Nuxt-приложения;
- JSON-based хранение данных;
- preview/render страница;
- стандартные block tools;
- стандартные inline tools;
- кастомные block tools;
- кастомные inline tools;
- block tunes;
- nested editors;
- экспорт JSON;
- local draft через `localStorage`;
- SCSS-архитектура;
- reusable field system для кастомных блоков;
- media gallery / slider block;
- sidebar navigation, строящаяся из JSON editor content на позднем этапе;
- code quality tooling и git hooks;
- поздняя локализация UI редактора;
- поздняя поддержка theme switching как отдельного UI-слоя приложения;
- поздние accessibility-улучшения, включая keyboard navigation.

### 4.2. Что не входит в scope

Следующие задачи **не входят в scope проекта** и не должны фигурировать как обязательные этапы реализации:

- авторизация;
- роли пользователей;
- права доступа;
- user management;
- полноценный backend API;
- production-ready upload backend;
- runtime SSR текущего проекта;
- полноценная page-management админка;
- многоязычный content model для самих данных контента;
- полноценный live locale switch без переинициализации Editor.js;
- полноценная theme engine уровня готовой CMS;
- обещание полностью бесшовного заполнения всего сложного Editor.js UI одной только клавиатурой уже на ранних этапах.

### 4.3. Что относится к поздним, но допустимым расширениям

Допускается реализация на поздних этапах:

- `zod`-валидации;
- `imask` и input masks;
- ручного `Import JSON`;
- продвинутых tunes;
- Fancybox deep-linking;
- sidebar-навигации по блокам;
- background/spoiler tunes;
- расширенной media-логики;
- collapse/expand поведения у вложенных пунктов sidebar navigation;
- локализации UI редактора;
- поддержки тёмной/светлой темы;
- использования community CSS-решений для Editor.js dark theme как основы для собственной темы;
- усиленной клавиатурной навигации и улучшений accessibility.
- расширения набора контентных блоков и Editor.js plugins после отдельного research/decision этапа.

## 5. Цели первой версии

Первая версия проекта должна обеспечивать:

- рабочий Editor.js внутри Nuxt-приложения;
- подключение стандартных block tools;
- подключение стандартных inline tools;
- сохранение результата в JSON;
- подстановку начальных данных из статического JSON или из `localStorage`;
- рендер сохранённого контента на отдельной preview/render-странице;
- экспорт JSON;
- reset локального draft;
- архитектурную основу для дальнейшего роста.

### Важно

- `Import JSON` не относится к обязательным функциям первой версии и должен рассматриваться как **поздний utility-сценарий**, а не как часть базового MVP.
- локализация и theme switching также не являются обязательной частью первой версии;
- keyboard navigation следует рассматривать как **поздний UX/accessibility этап**, а не как MVP-обязательство.

### 6.2. Дополнительные зависимости, допускаемые на поздних этапах

- `zod` — для валидации данных;
- `imask` — для input masks;

- community/official plugins Editor.js — по необходимости.

Примечания по поздним Editor.js plugins:

- backend-dependent plugins, например link preview tools с server-side metadata fetching, не являются обязательными для текущего frontend-only проекта;
- plugins, сохраняющие произвольный HTML, могут добавляться только после отдельного решения по trusted/admin-only режиму, sanitizer policy и renderer contract;
- для CTA/button-like сценариев предпочтителен typed custom block с явными вариантами отображения и target contract, если готовый plugin не даёт нужную модель данных.

### 6.3. Базовые зависимости проекта

#### Базовые зависимости приложения

Следует ориентироваться на актуальные версии:

- `nuxt`
- `typescript`
- `sass`
- `vue-i18n` — на позднем этапе локализации UI

Примечание:

- отдельная установка `vue-router` обычно не требуется, так как маршрутизация предоставляется Nuxt;
- Vue runtime поставляется вместе с Nuxt, но архитектурные решения editor-layer по-прежнему строятся вокруг Vue 3 Composition API.

#### Базовые зависимости Editor.js

Следует использовать актуальные совместимые версии:

- `@editorjs/editorjs`
- `@editorjs/header`
- `@editorjs/list`
- `@editorjs/image`
- `@editorjs/quote`
- `@editorjs/delimiter`
- `@editorjs/table`
- `@editorjs/embed`

Примечание:

- `@editorjs/paragraph` **не обязателен к установке**, так как Paragraph tool входит в Editor.js по умолчанию; отдельная установка требуется только если понадобится
  кастомная реализация Paragraph tool.
- Если потребуется отдельный checklist-сценарий, его следует анализировать уже после проверки актуального `@editorjs/list`, так как современный List Tool уже
  покрывает nesting, ordered/unordered и checklist-сценарии.

#### Базовые зависимости для inline tools

Следует ориентироваться на актуальные совместимые версии:

- `@editorjs/marker`
- `@editorjs/underline`
- `@editorjs/inline-code` — необязательный, более поздний
- сторонний plugin для `strikethrough` — сначала попытаться использовать готовое решение, затем при необходимости реализовать свой

Желательные кандидаты для предварительной проверки:

- `editorjs-strikethrough`
- `@gabbydgab/editorjs-strikethrough`

Окончательный выбор должен быть сделан только после проверки совместимости с остальными inline tools и сохранением/повторной загрузкой контента.

#### Зависимости для media/viewer-сценариев

- `swiper`
- `@fancyapps/ui`

Примечание:

- Для современных версий Swiper следует учитывать, что пакет больше не поставляет `.scss`/`.less` исходники, а официальные стили идут как обычный CSS. Это не мешает
  использовать Swiper в проекте, но требует отдельного внимания при интеграции со SCSS-архитектурой. Импорт официальных стилей следует рассматривать как
  инфраструктурный CSS-слой, а не как replacement для module SCSS.

#### Зависимости для поздней локализации

- `vue-i18n`

Примечание:

- `vue-i18n` относится к уровню Nuxt/Vue-приложения / editor shell.
- Editor.js должен получать подготовленный словарь через `i18n.messages` в конфиге.
- Внутри кастомных tools/tunes не предполагается прямое использование Vue runtime, но необходимо использовать единый словарь строк, совместимый с выбранной локалью.

## 8. Общие архитектурные принципы

### 8.1. Архитектура должна быть заложена с самого начала

С самого начала проект должен иметь корректное разделение слоёв, но без преждевременного усложнения.

Это означает:

- правильное разделение `shared / admin / renderer` необходимо сразу;
- тяжёлые абстракции не должны внедряться заранее без подтверждённой необходимости;
- архитектура должна расширяться по мере появления реальной сложности.

### 8.2. Базовый приоритет реализации

Приоритет реализации должен быть следующим:

1. Архитектурный каркас.
2. Shared-слой и registry блоков.
3. Источник данных и базовый editor shell.
4. Code quality tooling (`prettier`, `stylelint`, `husky`).
5. Стандартные block tools.
6. Стандартные inline tools.
7. Renderer.
8. Export / Reset draft UX.
9. Базовые tunes.
10. Plain field system.
11. Первый простой custom block.
12. Первый nested editor scenario.
13. Composite blocks.
14. Обязательные rich fields.
15. Кастомные inline tools.
16. Обязательный media gallery / slider block.
17. Sidebar navigation из JSON.
18. Поздние utility-сценарии (`Import JSON`).
19. Валидация.
20. Маски.
21. Локализация UI редактора.
22. Поддержка light/dark theme.
23. Клавиатурная навигация и accessibility polish.
24. Некритичные улучшения.
25. Расширение набора контентных блоков и plugins.

### 8.3. Принцип кастомизации

Кастомизация должна начинаться только после того, как:

- понятна базовая JSON-модель;
- стабильно работает `save/load`;
- проверена совместимость стандартных block tools и inline tools;
- готов renderer для стандартных блоков.

### 8.4. Inline tools как общий принцип

Все доступные inline tools — стандартные и кастомные — должны быть включены в **максимально возможном количестве совместимых block tools**.

Это означает:

- если конкретный block tool или nested editor технически поддерживает `inlineToolbar`, он должен быть включён;
- исключения должны быть явными, обоснованными и зафиксированными;
- inline tools должны быть доступны в paragraph, header, list и во всех rich field scenarios, где это имеет смысл.

### 8.5. Принцип поздних улучшений

Поздние улучшения не должны разрушать или перепроектировать базовую модель данных.

Это особенно важно для:

- локализации;
- theme switching;
- sidebar navigation;
- validation layer;
- media viewer behavior;
- keyboard navigation improvements.

## 9. Требования к структуре проекта

Проект должен использовать современную структуру Nuxt 4. Не следует сохранять устаревший layout с дополнительной верхнеуровневой директорией src/, если для этого нет
отдельной подтверждённой причины. Корнем приложения должен считаться корень репозитория с nuxt.config.ts.

Рекомендуемая структура проекта:

```text
/

    app.vue
    pages/
      index.vue
      editor.vue
      preview.vue
    components/
    composables/
    utils/
    layouts/
    plugins/
    styles/

  content/
    default-page.json

  i18n/
    locales/
      ru.ts
      en.ts
    editor/
      ru.ts
      en.ts

  editor/
    shared/
      types/
      entities/
      blocks/
      registry/
      constants/
      navigation/
    admin/
      core/
      config/
      tools/
        blocks/
        inline/
      tunes/
      fields/
      helpers/
      styles/
    renderer/
      components/
        blocks/
        navigation/
      helpers/
      types/

  shared/
    types/
    utils/
    constants/

  public/
  server/

  nuxt.config.ts
  package.json
  tsconfig.json
  app.config.ts
```

### 9.0. Общий подход к Nuxt 4-структуре

Необходимо придерживаться современных подходов **Nuxt 4**:

- не использовать отдельную верхнеуровневую директорию `src/` как основной layout проекта без отдельной причины;
- использовать `app/` как основной слой приложения;
- размещать страницы в `app/pages/`;
- размещать app-level компоненты, composables, utils, plugins и styles внутри `app/`;
- оставлять предметную подсистему редактора в отдельных корневых директориях (`editor/`, `content/`, `i18n/`, `shared/`), если это сохраняет архитектурную ясность;
- не жертвовать разделением `editor/shared`, `editor/admin`, `editor/renderer` ради слепого следования auto-import convention.

Nuxt 4 должен задавать современную инфраструктурную структуру проекта, но не размывать предметную архитектуру editor-layer.

### 9.1. Смысловое разделение слоёв

#### `editor/shared`

Содержит всё, что используется и в editor-layer, и в renderer-layer:

- типы данных;
- registry блоков;
- block data types;
- entity types;
- JSON-модели;
- общие константы;
- helper-типизацию для navigation tree.

#### `editor/admin`

Содержит всё, что относится к редактированию:

- создание editor config;
- block tools;
- inline tools;
- tunes;
- field builders;
- nested editor helpers/managers;
- editor infrastructure.

#### `editor/renderer`

Содержит всё, что относится к рендеру JSON:

- Vue-компоненты блоков;
- mapping `block.type -> component`;
- renderer utilities;
- renderer types;
- sidebar/navigation rendering.

#### `i18n`

Поздний, но заранее допустимый слой для локализации UI приложения и editor configuration:

- словари приложения;
- словари editor UI;
- mapping locale -> Editor.js i18n messages.

### 9.2. Правило структуры компонентов

Если компоненту требуются модульные стили, предпочтительная структура должна быть следующей:

```text
ComponentName/
  ComponentName.vue
  ComponentName.module.scss
```

Это правило особенно рекомендуется для:

- renderer-блоков;
- editor-shell компонентов;
- сложных UI-компонентов;
- блоков с собственной разметкой и логикой.

Для очень простых компонентов без собственных стилей отдельная директория не является обязательной, но желательна единообразная структура.

## 10. Требования к SCSS-архитектуре

### 10.1. Основной подход

Все Nuxt/Vue-компоненты должны в первую очередь стилизоваться через **модульные SCSS-файлы**.

Формат подключения стилей:

```vue

<style module lang="scss" src="./TwoColumns.module.scss" />
```

Этот подход обязателен для:

- обычных Nuxt/Vue-компонентов;
- renderer-компонентов блоков;
- страниц;
- editor-shell компонентов, если они реализованы как Vue-компоненты.

### 10.2. Подключение глобальных SCSS-зависимостей

Необходимо использовать глобальное подключение SCSS entrypoint через `nuxt.config.ts`, чтобы:

- переменные;
- функции;
- миксины;
- media helpers;
- общие SCSS-модули

не импортировались вручную в каждом `.module.scss`-файле.

Следует исходить из сценария, в котором глобальный SCSS entrypoint подмешивается через `additionalData` и/или соответствующую Vite-конфигурацию внутри Nuxt.

### 10.3. Базовая структура SCSS

За основу следует взять полноценную структуру SCSS-слоя с разделением на:

- `styles/utils/`
- `styles/utils/typography/`
- `styles/components/`
- `styles/_main.scss`
- `styles/_global.scss`
- `styles/common.scss`
- `styles/css-variables.scss`
- `styles/utilities.scss`

### 10.4. Глобальный SCSS-слой

Глобальный SCSS-слой должен содержать:

- переменные;
- функции;
- миксины;
- media helpers;
- CSS variables;
- utilities;
- reset/common rules.

### 10.5. Модульный SCSS-слой

Модульный SCSS-слой должен использоваться для стилизации конкретных Vue-компонентов.

### 10.6. Дополнительный global editor-specific слой

Допускается наличие дополнительных файлов наподобие:

- `editor-renderer.scss`
- `editor-admin.scss`

как **резервного вспомогательного слоя**, если для некоторых editor-related сущностей предпочтительнее использовать глобальные классы вместо модульных.

Важно:

- этот слой не является основным способом стилизации renderer-компонентов;
- renderer-компоненты всё равно должны в первую очередь использовать module SCSS;
- editor-specific global files допустимы как инфраструктурный слой и как запасной вариант.

### 10.7. Анимации и интеракции

Необходимо принять единое правило: все базовые интеракции и анимации должны иметь длительность **0.3s**.

Это относится к:

- hover-состояниям;
- toggle-переключателям;
- открытию/закрытию select;
- подсветке фокуса;
- появлению ошибок;
- служебным editor UI-переходам;
- collapse/expand поведениям в поздних сценариях sidebar navigation.

Следует использовать единый SCSS-токен перехода.

### 10.8. Подготовка к теме

Даже если dark/light theme не реализуется на ранних этапах, SCSS-архитектура должна по возможности опираться на токены и CSS variables.

Не следует без необходимости хардкодить цвета внутрь custom tools и editor UI.

## 11. Требования к модели данных

### 11.1. Registry-подход

В проекте должен использоваться единый registry блоков.

Registry должен быть единым источником истины для:

- editor-layer;
- renderer-layer;
- типизации блока по `type`;
- типизации `data` конкретного блока;
- типизации renderer props.

### 11.2. Discriminated union

Все блоки должны быть представлены как discriminated union по `type`.

Следует использовать общий тип блока, выводимый из registry.

### 11.3. Разделение типов

Не допускается сваливание всех типов в один большой файл.

Типы должны быть разделены минимум на:

- editor output types;
- registry types;
- block data types;
- entity types;
- navigation types.

### 11.4. JSON-модель

В качестве основной модели editor output необходимо использовать JSON-структуру, совместимую с `OutputData` Editor.js.

Допустимы alias-типы, но не следует создавать вторую конкурирующую модель без необходимости.

### 11.5. Локализация и тема не должны ломать модель данных

Ни локализация UI редактора, ни theme switching не должны требовать изменения content JSON schema.

Это отдельные слои приложения, а не свойства самих editor blocks.

### 11.6. Требование к alt для изображений

Во всех image-based сценариях необходимо предусматривать возможность задания `alt` как части данных.

Это относится к:

- одиночным изображениям в custom blocks;
- media items в gallery/slider block;
- кастомным image fields;
- любым будущим image-based composite blocks.

Для video media `alt` не требуется, но должны быть предусмотрены caption и/или description, если это предусмотрено сценарием блока.

## 12. Источник данных и хранение

### 12.1. Источник данных по умолчанию

Необходимо использовать статический JSON-файл проекта:

- `content/default-page.json`

### 12.2. Draft-хранение

Во время редактирования актуальный черновик должен сохраняться в `localStorage`.

### 12.3. Приоритет источников данных

При открытии редактора и preview необходимо использовать следующий приоритет:

1. Draft из `localStorage`, если он существует.
2. `default-page.json`, если draft отсутствует.

### 12.4. Обязательные действия первой версии

Для первой версии обязательно реализовать:

- `Export JSON`
- `Reset draft`
- disabled state для `Export JSON`

### 12.5. Поведение Export JSON

`Export JSON` должен:

- брать актуальные данные редактора;
- экспортировать их в `.json` файл;
- быть disabled, если отсутствует локальный draft или отсутствуют изменения относительно исходного состояния.

### 12.6. Поведение Reset draft

`Reset draft` должен:

- удалять локальный draft из `localStorage`;
- сбрасывать editor state;
- возвращать редактор и preview к содержимому `default-page.json`.

### 12.7. Import JSON

`Import JSON` не относится к обязательным функциям первой версии.

Он может быть добавлен как поздний utility-сценарий и, если реализуется, должен:

- принимать `.json` файл от пользователя;
- проверять корректность файла как минимум на базовом уровне;
- загружать содержимое в editor state;
- записывать его в `localStorage` как текущий draft.

### 12.8. Базовая валидация Import JSON

Если `Import JSON` реализуется, на первом шаге необходима только базовая валидация:

- файл читается как JSON;
- верхний уровень является объектом;
- у объекта есть `blocks`;
- `blocks` — массив.

Глубокая проверка структуры всех блоков относится к поздним этапам.

### 12.9. Браузерная запись в проектный JSON

Браузер не должен пытаться напрямую обновлять JSON-файл проекта как обычный файл исходного кода.

Обновление проектного JSON должно происходить вручную: через экспорт и дальнейшее сохранение в репозиторий.

## 13. Требования к editor-layer

### 13.1. Editor config

Необходимо реализовать фабрику конфигурации редактора, например `createEditorConfig`, которая:

- принимает стартовые данные;
- подключает block tools;
- подключает inline tools;
- подключает tunes;
- задаёт `onChange`;
- возвращает единый конфиг Editor.js.

### 13.2. Базовый editor-shell

Необходимо реализовать editor-shell как Nuxt/Vue-компонент, который:

- создаёт и уничтожает Editor.js instance;
- получает initial data;
- обрабатывает save/update;
- показывает служебные действия редактора.

### 13.3. Прямое использование `window.EditorJS`

Необходимо избегать зависимости от глобального `window.EditorJS`, если используется модульная установка через npm.

### 13.4. Инициализация без `setTimeout`

Необходимо стремиться к предсказуемому lifecycle.

Если вложенные editor instances потребуются позже, их инициализация должна строиться на явном lifecycle-менеджменте, а не на случайных `setTimeout`.

### 13.5. Подготовка к локализации

Тексты editor UI не должны без необходимости жёстко хардкодиться внутри tools/tunes/field builders.

Следует заранее учитывать, что позднее строки могут быть вынесены в словари.

### 13.6. Подготовка к темизации

Editor shell и custom tools должны по возможности использовать theme-friendly class names и CSS variables.

### 13.7. Подготовка к accessibility и keyboard navigation

Необходимо по возможности избегать решений, которые полностью блокируют естественный `tab`-порядок и затрудняют навигацию без мыши.

Это особенно важно для:

- plain field wrappers;
- toggle/select/button UI;
- управления списками карточек/слайдов;
- sidebar navigation;
- поздних composite blocks.

При этом не требуется обещать полное keyboard-only покрытие уже на ранних этапах.

## 14. Стандартные block tools: обязательный первый функциональный этап

### 14.1. Цель этапа

Поднять рабочий Editor.js на стандартных блоках и получить корректный JSON output.

### 14.2. Базовый набор block tools

На старте должны быть подключены:

- Paragraph
- Header
- List
- Image
- Quote
- Delimiter
- Table
- Embed

### 14.3. Требования к List tool

В проекте не должно планироваться два независимых пользовательских плагина — “одномерный список” и “многомерный список”.

Следует использовать **один основной list tool**, который:

- на первом этапе работает как ordered/unordered list;
- на следующем этапе расширяется до nested list;
- не дублируется вторым отдельным list-плагином без крайней необходимости.

### 14.4. Что необходимо проверить

После реализации этапа необходимо проверить:

- редактор успешно инициализируется;
- все подключённые block tools отображаются;
- блоки можно добавлять, редактировать, удалять и перемещать;
- `save()` возвращает валидный JSON;
- сохранённый JSON можно повторно загрузить в editor;
- типы `type` в output совпадают с ключами tools конфигурации.

### 14.5. Критерии готовности этапа

Этап считается завершённым, если:

- редактор стабильно работает на стандартных блоках;
- save/load проходят без потери данных;
- нет необходимости в кастомных block tools для запуска базовой версии.

## 15. Стандартные inline tools: обязательный второй функциональный этап

### 15.1. Цель этапа

Подключить и проверить стандартные inline tools до начала кастомизации.

### 15.2. Базовый набор inline tools

На старте должны быть подключены:

- bold
- italic
- link
- underline
- marker (для фона текста)
- strikethrough

`inline code` допускается добавить позже как необязательный инструмент.

### 15.3. Требование к strikethrough

`Strikethrough` должен считаться обязательным inline tool проекта.

Предпочтительный порядок реализации:

1. Сначала попытаться использовать готовый сторонний совместимый plugin.
2. Если сторонний plugin не проходит проверку стабильности и совместимости, реализовать собственный strikethrough tool.

### 15.4. Доступность inline tools

Все стандартные inline tools должны быть включены в максимально возможном количестве совместимых блоков, включая:

- paragraph;
- header;
- list;
- rich field scenarios на nested Editor.js.

### 15.5. Проверка совместимости

После реализации необходимо проверить комбинации:

- bold + italic
- bold + link
- underline + link
- marker + link
- marker + underline
- strikethrough + bold
- strikethrough + italic
- strikethrough + link
- strikethrough + underline
- strikethrough + marker
- bold + italic + link
- marker + bold + link

Также необходимо проверить:

- повторное открытие ранее сохранённого контента;
- корректность рендера комбинаций;
- отсутствие потери разметки после повторного `save()`.

### 15.6. Inline tools в списках

Для list tool и nested list scenarios необходимо явно включить `inlineToolbar`, если требуется поддержка inline formatting в пунктах списка.

### 15.7. Критерии готовности этапа

Этап считается завершённым, если:

- стандартные inline tools работают стабильно;
- базовые комбинации инструментов не ломаются;
- сохранение и повторная загрузка не искажают контент;
- inline tools доступны в максимально возможном количестве совместимых tool-сценариев.

## 16. Renderer: обязательный ранний этап

### 16.1. Цель этапа

Реализовать отдельный renderer JSON-контента на Vue-компонентах.

### 16.2. Требования к renderer

Renderer должен:

- принимать JSON editor output;
- по `block.type` определять нужный Vue-компонент;
- передавать в компонент типизированные `data`;
- корректно рендерить стандартные block tools.

### 16.3. Стили renderer-компонентов

Renderer-компоненты должны в первую очередь использовать **module SCSS**.

Отдельный глобальный `editor-renderer.scss` допускается оставить как резервный слой, но не как основной способ стилизации renderer.

### 16.4. Проверка

Необходимо проверить:

- корректный рендер всех стандартных block tools;
- совпадение структуры JSON и renderer-модели;
- отсутствие зависимости renderer от editor runtime.

### 16.5. Критерии готовности этапа

Этап считается завершённым, если:

- preview-страница корректно рендерит JSON;
- renderer отделён от editor-layer;
- компоненты renderer используют типизированные данные.

## 17. Block Tunes: первый этап кастомизации

### 17.1. Общий подход

Block Tunes должны использоваться для дополнительных метаданных блока, а не для внутренней логики конкретного block tool.

### 17.2. Приоритет tunes

Порядок реализации:

1. `AnchorTune`
2. `SpacingTune`
3. `LabelTune` / `TitleTune`
4. `BackgroundTune` — поздний этап
5. `SpoilerTune` — поздний этап

### 17.3. AnchorTune

Должен хранить значение якоря блока и использоваться для скролла по ссылкам вида `#anchor`.

### 17.4. SpacingTune

Должен быть основным решением для задания верхних и/или нижних отступов блока.

### 17.5. LabelTune / TitleTune

Должен хранить label для навигации по странице и использоваться для sidebar-навигации.

### 17.6. BackgroundTune

Допускается как поздний tune для выбора фонового режима блока.

### 17.7. SpoilerTune

Допускается как поздний tune для скрываемых/раскрываемых блоков.

### 17.8. Проверка

После реализации каждого tune необходимо проверить:

- сохранение состояния tune;
- повторную загрузку состояния;
- применение данных tune в renderer;
- отсутствие влияния на блоки без tune-data.

### 17.9. Критерии готовности этапа

Tune считается реализованным, если:

- значение можно изменить в editor UI;
- значение сохраняется в JSON;
- renderer корректно использует значение;
- блок без заданного tune продолжает работать штатно.

## 18. Поля в custom blocks

### 18.1. Общий подход

В проекте должны поддерживаться два типа полей:

- `plain fields`
- `rich fields`

### 18.2. Plain fields

К plain fields относятся:

- text input
- textarea
- select
- radio group
- toggle
- image/url/file selectors

На ранних этапах необходимо реализовать именно plain fields.

### 18.3. Toggle

Checkbox в editor UI должен иметь внешний вид toggle-переключателя, при этом на уровне данных он остаётся обычным boolean.

### 18.4. Rich fields

Rich fields должны быть предусмотрены архитектурно **с самого начала** и **обязательно реализованы позднее**.

Rich fields не являются опциональной возможностью “только если понадобится”.

К rich fields относятся:

- nested paragraph field;
- nested header field;
- другие single-purpose nested editors.

### 18.5. Архитектурное требование

Одно и то же логическое поле должно допускать в будущем две реализации:

- как plain field;
- как rich field.

### 18.6. Ограничение plain fields

Обычные `input` и `textarea` не должны рассматриваться как носители inline tools.

Если полю нужны inline tools, оно должно быть реализовано как rich field на nested Editor.js.

### 18.7. Field wrappers

Для plain field system необходимо использовать единообразную структуру поля, включающую:

- field wrapper;
- label;
- control;
- optional hint;
- optional error.

Под field wrapper понимается обычная UI-обёртка поля, а не отдельная технология.

### 18.8. Медиа-поля в custom blocks

Для кастомных блоков загрузка и выбор изображений/видео должны реализовываться прежде всего через **отдельные media fields**, а не через nested Editor.js с
единственным image tool.

Nested Editor.js для image/video не должен считаться базовым способом работы с media в custom blocks.

### 18.9. Alt в media fields

Для всех image-based media fields должно быть предусмотрено явное поле `alt`.

Это относится к:

- одиночным image fields;
- image items в gallery/slider;
- image fields внутри custom blocks;
- будущим composite blocks, если они содержат изображения.

## 19. Первый custom block: только после tunes и plain fields

### 19.1. Цель этапа

Реализовать первый простой custom block без nested editors.

### 19.2. Обязательные тренировочные custom blocks

Первые простые custom blocks должны выбираться из следующего набора:

- CTA
- Section Intro
- Notice
- Banner
- SpacerBlock (только как дополнительный блок, если действительно понадобится)

### 19.3. Роли и примерные данные блоков

#### CTA

Назначение: блок призыва к действию.

Примерные поля:

- title
- description
- buttonText
- buttonUrl
- theme / variant
- optional image

#### Section Intro

Назначение: вводный блок секции.

Примерные поля:

- title
- subtitle / short text
- optional aside
- optional meta/tune data

#### Notice

Назначение: информационный или предупреждающий блок.

Примерные поля:

- title
- text
- type (`info | warning | success | error`)

#### Banner

Назначение: баннерный/hero-like блок.

Примерные поля:

- title
- description
- image
- buttonText
- buttonUrl
- theme
- mediaPosition

#### SpacerBlock

Назначение: самостоятельный layout-gap блок.

Примерные поля:

- spacing preset

### 19.4. Требования к custom block

Каждый custom block должен иметь:

- собственный data type;
- renderer component;
- tool class;
- save/load сценарий;
- при необходимости — tune support;
- при необходимости — простые поля.

### 19.5. Что проверить

- блок можно создать;
- блок можно удалить;
- блок корректно сохраняет данные;
- блок корректно восстанавливается из JSON;
- renderer умеет его отрисовывать.

### 19.6. Критерии готовности этапа

Этап считается завершённым, если как минимум один custom block полностью проходит цикл `create -> edit -> save -> reload -> render`.

## 20. Nested editors: отдельный этап

### 20.1. Общий принцип

Nested editors должны добавляться только после того, как:

- работают plain fields;
- реализованы tunes;
- есть хотя бы один простой custom block;
- отлажен базовый lifecycle editor.

### 20.2. Первый nested editor

Первым должен быть реализован **один простой custom block с одним nested editor**.

Не следует начинать сразу с `TwoColumns`.

### 20.3. Ограничение nested editors

Во вложенном editor instance должен быть жёстко ограничен набор tools.

Примеры:

- только `paragraph`;
- только `header`;
- `paragraph + list`.

### 20.4. Single-purpose nested editors

Необходимо предусмотреть режим nested editor, который используется как почти “одно поле”, например:

- rich paragraph field;
- rich header field.

### 20.5. Header-field стратегия

`Header` tool нативно поддерживает выбор уровня заголовка.

Следовательно, долгосрочное целевое решение для rich header field — вложенный editor на основе official `Header` tool, а не только комбинация `input + select`.

На раннем этапе допускается временная реализация `input + select(level)`, но это не должно считаться финальным целевым решением.

### 20.6. Ограничение одного блока

Если вложенный editor используется как одно rich-поле, необходимо своей логикой ограничивать или валидировать сценарий, в котором допускается только один block.

### 20.7. Проверка

- nested editor корректно создаётся;
- nested editor корректно уничтожается;
- save/load nested data работают стабильно;
- ограниченный whitelist tools соблюдается;
- renderer корректно рендерит вложенный контент.

### 20.8. Критерии готовности этапа

Этап считается завершённым, если существует хотя бы один custom block с корректно работающим single-purpose nested editor.

## 21. Rich fields: обязательный поздний этап

### 21.1. Общий принцип

После plain field system и первого nested editor scenario необходимо обязательно реализовать reusable rich fields.

### 21.2. Обязательные rich field scenarios

Необходимо реализовать как минимум:

- `RichParagraphField`
- `RichHeaderField`

### 21.3. Требования к rich fields

Rich fields должны:

- использовать nested Editor.js;
- иметь ограниченный whitelist tools;
- поддерживать inline tools;
- корректно сохраняться и восстанавливаться;
- рендериться как обычные типизированные поля блока.

### 21.4. Что проверить

- rich field создаётся корректно;
- rich field стабильно работает как nested editor;
- inline tools доступны;
- save/load не искажают данные;
- renderer корректно рендерит содержимое.

### 21.5. Критерии готовности этапа

Этап считается завершённым, если reusable rich fields реализованы и могут использоваться внутри последующих custom blocks.

## 22. TwoColumns и другие composite blocks

### 22.1. Порядок

`TwoColumns` и другие сложные composite blocks должны реализовываться только после успешного этапа с одним nested editor и базовыми rich fields.

### 22.2. Требования к TwoColumns

`TwoColumns` должен хранить:

- layout variant;
- direction / reversed state;
- tune data;
- left content;
- right content.

`left` и `right` должны быть представлены как отдельные nested editor outputs.

### 22.3. Разрешённые инструменты во вложенных колонках

Для колонок должен использоваться ограниченный whitelist tools, например:

- paragraph
- header
- list

### 22.4. Проверка

- корректный save/load обеих колонок;
- отсутствие конфликтов toolbar;
- корректный renderer;
- стабильное поведение при удалении и перемещении блока.

## 23. Кастомные inline tools

### 23.1. Порядок реализации

Кастомные inline tools должны реализовываться только после стандартных inline tools и после появления стабильных nested editor scenarios.

### 23.2. Обязательные inline tools

Следует реализовать:

1. Стандартный `Marker` для цвета фона текста.
2. Кастомный inline tool для цвета текста.

### 23.3. Общий подход к кастомному inline tool

Кастомный tool для цвета текста должен:

- использовать собственную inline-обёртку;
- использовать собственный CSS class;
- не мутировать существующие `b`, `i`, `a`, `u` и другие inline-элементы;
- корректно сохраняться и восстанавливаться.

### 23.4. Доступность custom inline tools

Кастомный inline tool для цвета текста должен включаться в максимально возможном количестве совместимых block tools и rich field scenarios.

### 23.5. Проверка

Необходимо проверить комбинации:

- text color + bold
- text color + italic
- text color + link
- text color + underline
- text color + background marker
- text color + strikethrough
- text color + несколько стандартных inline tools одновременно

### 23.6. Критерии готовности этапа

Этап считается завершённым, если кастомный inline tool:

- работает в nested editor и/или основном editor;
- корректно сохраняется;
- корректно рендерится;
- не ломает комбинации со стандартными inline tools.

## 24. Media gallery / slider block: обязательный composite-блок

### 24.1. Общий принцип

Media gallery / slider block является обязательным кастомным блоком и должен быть реализован поэтапно.

### 24.2. Обязательное технологическое требование

Для всех слайдерных сценариев необходимо использовать **Swiper**.

Самописный слайдер не допускается как базовое решение.

### 24.3. Рекомендуемая стратегия

На первом этапе следует рассматривать **один общий media block** с режимом:

- `slider`
- `gallery`

Если позже различие по UX и структуре данных окажется слишком большим, допускается разделение на два независимых блока.

### 24.4. Определения режимов

- `gallery` — это **сетка карточек**.
- `slider` — это **перелистываемая последовательность карточек на Swiper**.

### 24.5. UI управления карточками/слайдами

Внутри block UI необходимо предусмотреть собственный интерфейс управления массивом карточек/слайдов.

Минимально должны быть доступны:

- кнопка `Добавить слайд` / `Добавить карточку`;
- удаление карточки;
- изменение порядка карточек;
- заполнение media fields карточки;
- заполнение caption;
- plain field для `alt` у изображений;
- nested rich description для карточки или слайда, если он предусмотрен текущим этапом.

### 24.6. Этап 1 media-block

На первом этапе у каждого слайда должен быть:

- один media item (`image | video`);
- caption;
- `alt` для изображений;
- optional rich description.

У блока должны быть настройки:

- `mode: slider | gallery`
- `galleryId`
- `enableFancybox`
- `syncUrlWithFancybox`

### 24.7. Этап 2 media-block

На втором этапе допускается расширение, при котором у одного слайда:

- несколько media items;
- одно общее description;
- более сложная внутренняя media-структура.

### 24.8. Fancybox grouping и URL

Необходимо предусмотреть:

- группировку media items по `galleryId`;
- возможность открытия конкретного элемента по URL;
- смену URL при перелистывании в Fancybox;
- восстановление состояния после перезагрузки страницы, если URL указывает на конкретный media item.

На раннем этапе допускается реализация через индекс, но долгосрочно предпочтительнее стабильный slide/media id.

### 24.9. Caption и description

Необходимо предусмотреть:

- caption для показа в карточке и/или Fancybox;
- optional rich description через nested Editor.js.

### 24.10. Тогглы и дополнительные поля блока

В блоке должны быть предусмотрены:

- toggle для `enableFancybox`;
- toggle для `syncUrlWithFancybox`;
- plain input для `galleryId` / viewer id.

### 24.11. Проверка

Необходимо проверить:

- render как `slider`;
- render как `gallery`;
- корректную работу Swiper;
- корректную работу Fancybox;
- корректную группировку по `galleryId`;
- корректную работу caption;
- корректную работу `alt`;
- корректный save/load JSON.

### 24.12. Критерии готовности этапа

Этап считается завершённым, если реализован минимум первый этап media gallery/slider block со стабильным save/load/render и viewer behavior.

## 25. List tool и nested list: обязательный поэтапный сценарий

### 25.1. Общий принцип

В проекте не требуется отдельный самостоятельный “одномерный list plugin”.

Нужен **один основной list tool**, который проходит два этапа развития.

### 25.2. Этап 1

List tool должен поддерживать:

- ordered list;
- unordered list;
- inline tools в пунктах списка.

На этом этапе список может использоваться как обычный одноуровневый список.

### 25.3. Этап 2

Тот же list tool должен быть доведён до nested list behavior:

- вложенные уровни;
- ordered/unordered на всех уровнях;
- проверка вложенности и UX переходов.

### 25.4. Поздняя исследовательская задача

После всех основных этапов допускается исследование сценария, в котором тип списка (`ordered/unordered`) задаётся отдельно для каждого уровня.

Это не относится к базовой обязательной реализации.

### 25.5. Проверка

Необходимо проверить:

- корректный save/load list data;
- корректный рендер списков;
- inline tools в элементах списка;
- переход от простого режима к nested режиму без смены пользовательского блока.

## 26. Sidebar navigation по контенту: поздний этап

### 26.1. Общий принцип

Sidebar navigation должна строиться **из JSON editor content**, а не путём сканирования итогового DOM после рендера страницы.

JSON является источником истины для построения navigation-структуры.

### 26.2. Базовые источники данных navigation

На позднем этапе sidebar navigation должна использовать:

- `AnchorTune`
- `LabelTune`
- при необходимости уровень заголовка `Header.level`

### 26.3. Этап 1: одномерная navigation

На первом этапе необходимо реализовать:

- построение плоского списка пунктов меню из JSON;
- использование `label + anchor`;
- рендер sidebar как обычного списка ссылок.

### 26.4. Этап 2: вложенная navigation

На втором этапе необходимо реализовать:

- определение вложенности пунктов;
- построение navigation tree;
- использование уровней заголовков как основного сигнала вложенности.

### 26.5. Позднее поведение

После реализации tree-navigation допускается добавить:

- collapse / expand;
- accordion behavior;
- дополнительные опции отображения.

### 26.6. Что не требуется на текущем этапе

Sidebar navigation не должна зависеть от будущих page-level settings следующего fullstack-проекта.

Однако архитектурно её следует строить так, чтобы позднее можно было передавать внешние `sidebar options` на уровне страницы.

### 26.7. Проверка

- navigation строится из JSON, а не из DOM;
- flat navigation работает стабильно;
- anchor navigation корректна;
- tree navigation не ломает базовый flat-сценарий.

## 27. Валидация и masks: поздний этап

### 27.1. Приоритет

Валидация не относится к ранним этапам и должна появляться ближе к завершению базовой редакторной функциональности.

### 27.2. Библиотеки

Допускается использовать:

- `zod` — для схем и декларативной валидации;
- `imask` — для input masks.

### 27.3. Уровни валидации

#### Plain fields

Ошибки должны отображаться под конкретным полем.

#### Nested editors / rich fields

Ошибки должны отображаться под конкретным nested editor как под цельным полем.

#### Внутренние блоки nested editor

Адресная валидация внутренних paragraph/header/list block-элементов не является обязательной для первой версии и может быть отложена.

### 27.4. Минимальная валидация

На первом шаге валидации достаточно:

- required;
- min/max length;
- email;
- url;
- phone pattern.

### 27.5. Полная валидация

Более сложная валидация структуры блоков, nested data и поздние masks относятся к одному из последних этапов.

### 27.6. Критерии готовности этапа

Этап считается завершённым, если:

- plain fields поддерживают базовые ошибки;
- nested editor fields поддерживают ошибки на уровне поля;
- ошибки отображаются предсказуемо;
- сохранение и экспорт можно ограничить при критичных ошибках.

- static files из проекта;
- public assets;
- URL;
- при необходимости временные blob/data URLs.

### 28.2. Видео

На раннем этапе необходимо использовать:

- embed URL;

Caption не должен автоматически подменять собой `alt` без явного решения.

### 28.5. Что не требуется

Не требуется реализовывать полноценный upload flow до появления бэкенда.

## 29. Локализация UI редактора: поздний этап

### 29.1. Общий принцип

Локализация в рамках текущего проекта относится к **UI редактора и оболочки приложения**, а не к content JSON.

### 29.2. Подход

Необходимо использовать:

- `vue-i18n` на уровне Nuxt/Vue-приложения;
- `Editor.js i18n.messages` на уровне конфигурации Editor.js.

### 29.3. Минимальный scope локализации

На позднем этапе необходимо предусмотреть минимум две локали:

- `ru`
- `en`

Локализации должны подлежать:

- editor shell UI;
- названия tools;
- labels и placeholders custom fields;
- тексты toggle/select/button внутри custom blocks;
- tune labels;
- прочие editor-related UI strings.

### 29.4. Что не требуется

Не требуется:

- многоязычный content model;
- хранение нескольких языковых версий одного контента;
- безусловный live switch языка без переинициализации editor instance.

### 29.5. Допустимый сценарий смены языка

Если позднее понадобится смена языка, допустим сценарий:

- сохранить editor state;
- уничтожить editor instance;
- создать editor instance заново с новой локалью.

### 29.6. Проверка

- UI редактора корректно отображается минимум на двух языках;
- контент не теряется при переинициализации;
- словари Editor.js и словари приложения согласованы.

### 29.7. Критерии готовности этапа

Этап считается завершённым, если editor UI доступен минимум на русском и английском языках без изменения content schema.

## 30. Поддержка light/dark theme: поздний этап

### 30.1. Общий принцип

У Editor.js нет готовой полноценной theme engine, на которую можно безусловно опереться как на встроенное решение.

Поддержка темы в текущем проекте должна рассматриваться как **отдельный UI-слой приложения**, который вручную темизирует:

- editor shell;
- базовый Editor.js UI;
- custom tools;
- custom fields;
- поздние composite blocks.

### 30.2. Допустимая база

Допускается:

- использовать готовые community CSS-файлы и сниппеты для dark mode Editor.js как основу;
- адаптировать их под собственную дизайн-систему;
- сочетать их с собственными CSS variables и токенами.

### 30.3. Scope темы

Поддержка темы, если реализуется, должна покрывать:

- editor shell;
- базовый Editor.js UI;
- кастомные tools;
- кастомные fields;
- toggle/select/button wrappers;
- по возможности поздние media/viewer сценарии.

### 30.4. Что не предполагается автоматически

Не следует рассчитывать, что:

- дефолтные инструменты Editor.js автоматически подхватят dark mode;
- custom tools автоматически будут корректно темизироваться без собственной работы;
- достаточным будет только один внешний CSS без интеграции в токены проекта.

### 30.5. Архитектурный подход

Предпочтительный путь:

- светлая тема является базовой;
- позднее добавляется dark theme;
- переключение темы относится к уровню приложения / editor shell;
- Editor.js UI темизируется через CSS variables и overrides.

### 30.6. Проверка

Необходимо проверить:

- переключение shell UI;
- корректность отображения базовых Editor.js control states;
- корректность отображения custom tools;
- читаемость текста, кнопок, бордеров, hover/focus состояний.

### 30.7. Критерии готовности этапа

Этап считается завершённым, если редактор и ключевые кастомные сценарии визуально читаемы и функционально корректны в обеих темах.

## 31. Клавиатурная навигация и accessibility polish: самый поздний этап

### 31.1. Общий принцип

Удобная навигация при помощи клавиатуры является желательной частью UX, но из-за сложности Editor.js и большого количества динамических элементов относится к *
*одному из самых поздних этапов**.

### 31.2. Что уже можно ожидать от базовой версии

На ранних этапах достаточно не ломать естественные сценарии браузера там, где это возможно:

- `Tab` между plain fields;
- `Enter` / `Space` на кнопках и toggle;
- предсказуемый фокус в editor shell;
- отсутствие искусственных focus-traps без необходимости.

### 31.3. Что не следует обещать на ранних этапах

Не следует обещать, что:

- весь Editor.js UI будет полностью и безупречно управляться одной только клавиатурой уже в MVP;
- все сложные dropdown/toolbox/tune scenarios будут одинаково удобны без дополнительной доработки;
- сам Editor.js гарантирует безусловно идеальную tab navigation для всех custom scenarios.

### 31.4. Поздний scope keyboard navigation

На позднем этапе необходимо по возможности обеспечить:

- заполнение plain fields без мыши;
- управление toggle/select/button wrappers с клавиатуры;
- управление списком карточек/слайдов с клавиатуры;
- предсказуемый tab-order внутри composite blocks;
- поздние hotkeys для частых действий, если это действительно улучшает UX;
- отсутствие критичных фокусных ловушек между editor shell и соседними полями страницы.

### 31.5. Минимальные требования к реализации

Следует предусмотреть:

- видимые focus-состояния;
- отсутствие скрытия важного focus outline без замены;
- корректное использование нативных `button`, `input`, `select`, `textarea`, где это возможно;
- избегание лишних `div`-элементов в роли интерактивных контролов без keyboard handling.

### 31.6. Проверка

Необходимо проверить:

- можно ли пройти основные plain field сценарии только клавиатурой;
- работают ли toggle и основные editor action buttons без мыши;
- не застревает ли фокус в critical сценариях;
- не ломает ли поздняя keyboard-polish базовый editor UX.

### 31.7. Критерии готовности этапа

Этап считается завершённым, если основные editor-related формы и ключевые поздние custom UI-сценарии можно пройти предсказуемо и удобно с клавиатуры, без критичных
focus traps.

## 32. Важные технические уточнения и потенциальные ошибки понимания

### 32.1. Paragraph tool

Paragraph tool встроен в Editor.js по умолчанию. Его не нужно устанавливать отдельно, если не требуется специальная кастомизация.

### 32.2. Built-in inline tools

У Editor.js по умолчанию встроены только `bold`, `italic` и `link` как built-in inline tools. `marker`, `underline`, `inline-code` и `strikethrough` требуют
отдельных plugins или кастомной реализации.

### 32.3. List strategy

Для текущей архитектуры не следует планировать отдельный “simple list plugin” и отдельный “nested list plugin” как две разные пользовательские сущности. Текущая
стратегия с одним основным List Tool и поэтапным расширением выглядит корректно.

### 32.4. Single-purpose nested editors

Editor.js является block editor, а не single-field rich text editor. Сценарий “ровно один block в nested editor” реализуем, но потребует собственной логики
ограничения и/или валидации.

### 32.5. Image tool без backend

Official Image Tool существует, но полноценный upload workflow без backend по сути нужно будет эмулировать через URL/static files/custom uploader callbacks. Для
custom blocks это не отменяет предпочтительность отдельных media fields.

### 32.6. Alt support

Поддержку `alt` не следует считать автоматически закрытой за счёт official image ecosystem. Для custom media fields и composite blocks ответственность за `alt`
должна быть зафиксирована явно в собственной data model.

### 32.7. Sidebar navigation

Построение sidebar navigation из JSON реализуемо и является правильным направлением. Сбор navigation из DOM после рендера следует считать менее предпочтительным.

### 32.8. Swiper styles

Swiper в современных версиях поставляет официальные стили как CSS. Это не запрещает его использование, но требует осознанного включения в общую стилевую систему
проекта.

### 32.9. Strikethrough stability

Strikethrough следует сначала пробовать как готовый plugin, но необходимо быть готовым к собственной реализации, если готовое решение будет нестабильным.

### 32.10. Localization approach

Использование `vue-i18n` на уровне приложения и `i18n.messages` на уровне Editor.js не противоречит подходам самого Editor.js. Это и есть корректная совместимая
схема локализации UI редактора.

### 32.11. Theme expectations

Не следует ожидать, что Editor.js сам по себе обеспечивает полноценное переключение темы для всех official и custom tools. Тему нужно проектировать как слой
приложения.

### 32.12. Keyboard navigation complexity

Клавиатурная навигация является полезной целью, но Editor.js как сложный JS-widget может создавать дополнительные сложности для tab-order и focus management. Поэтому
этот сценарий обоснованно вынесен в самый конец.

## 35. Реализация проекта сразу на Nuxt и static/prerender модель

### 35.1. Актуальное архитектурное решение

Текущий проект должен **сразу** реализовываться на **Nuxt**.

Не должно существовать отдельной промежуточной стадии, на которой проект сначала собирается как самостоятельный проект на `Vue + Vite`, а затем позднее переносится
на Nuxt.

Любые упоминания `Vue + Vite` из более раннего обсуждения должны рассматриваться только как исторический контекст, а не как отдельный этап текущего плана.

### 35.2. Что это означает practically

Это означает следующее:

- базовый репозиторий должен сразу создаваться как Nuxt-проект;
- страницы редактора и preview должны сразу проектироваться в Nuxt-контексте;
- editor-shell, renderer-shell и доменный editor-layer должны сразу жить внутри Nuxt-кодовой базы;
- ранее согласованные editor-specific решения должны внедряться непосредственно в Nuxt-код, без промежуточной миграционной стадии.

### 35.3. Что сохраняется без изменений

Несмотря на выбор Nuxt, не должны меняться уже согласованные editor-specific решения:

- registry-подход;
- discriminated union;
- разделение `editor/shared`, `editor/admin`, `editor/renderer`;
- JSON workflow;
- plain/rich field model;
- tunes;
- nested editors;
- media gallery / slider block;
- sidebar navigation из JSON;
- локализация editor UI;
- theme layer;
- поздние accessibility-улучшения.

### 35.4. Static/prerender требования

Текущий проект должен использовать **Nuxt** не как runtime SSR-приложение, а как **static/prerender deployment target**.

Это означает:

- текущий проект не должен требовать наличия постоянно работающего Node.js-сервера;
- маршруты текущего проекта должны быть совместимы со static hosting;
- Editor.js и связанные с ним client-side сценарии должны корректно инициализироваться в статически собранном Nuxt-приложении;
- использование Nuxt должно уменьшать будущий объём миграции в следующий fullstack-проект, а не увеличивать его.

### 35.5. Требования к маршрутам и client-only интеграциям

На уровне текущего проекта достаточно предусмотреть как минимум:

- главную/служебную страницу проекта;
- страницу редактора;
- страницу preview/render;
- при необходимости демонстрационные страницы для отдельных сценариев сложных блоков.

Следует учитывать, что Editor.js является client-side редактором и должен инициализироваться только в браузерной среде.

Это означает:

- Editor.js instance не должен инициализироваться в server/runtime-контексте;
- интеграция должна быть организована через client-safe lifecycle;
- доступ к `window`, `document`, selection API и аналогичным браузерным сущностям должен выполняться только в корректных client-сценариях;
- editor-shell должен быть совместим с hydration и static/prerender моделью Nuxt.

### 35.6. Требования к Nuxt-конфигурации

В `nuxt.config.ts` должны быть заранее продуманы:

- статический/prerender режим сборки;
- корректный `app.baseURL` для GitHub Pages subpath, если сайт публикуется не в корень домена;
- подключение глобального SCSS entrypoint;
- client-safe интеграция browser-only зависимостей;
- отсутствие инфраструктурных решений, которые завязаны на runtime SSR текущего проекта.

## 36. GitHub Pages, GitHub Actions и минимальная Nuxt-инфраструктура

### 36.1. Общий принцип CI/CD

Проект должен быть настроен так, чтобы при каждом push в основную ветку репозитория выполнялись:

- установка зависимостей;
- статическая сборка Nuxt-проекта;
- публикация результата на GitHub Pages.

Допускается также ручной запуск workflow через `workflow_dispatch`.

### 36.2. Базовые требования к workflow

Workflow должен:

- использовать GitHub Actions;
- храниться в `.github/workflows/`;
- собирать проект в CI;
- публиковать статический артефакт;
- использовать GitHub Pages deployment flow.

При этом сам workflow не должен подменять архитектурные требования проекта: он лишь автоматизирует публикацию уже корректно собранного static/prerender
Nuxt-приложения.

### 36.3. Что необходимо проверить после настройки CI/CD

После настройки GitHub Actions и GitHub Pages необходимо проверить:

- workflow запускается автоматически по push;
- build проходит без ручных шагов;
- опубликованный сайт открывается корректно;
- базовый путь репозитория не ломает роутинг;
- статические assets загружаются по корректным путям;
- editor и preview страницы открываются на опубликованной версии сайта.

### 36.4. Рекомендуемая минимальная Nuxt-инфраструктура

Даже если часть каталогов фактически появится позже, архитектурно следует заранее предусмотреть:

- `app/` как основной слой страниц и клиентского UI;
- `editor/` как доменно-редакторный слой;
- `content/` или `public/content/` как источник статического demo JSON;
- `styles/` как глобальный SCSS-слой;
- `i18n/` как поздний, но заранее допустимый слой локализации.

На раннем этапе желательно иметь следующие страницы:

- главную/служебную страницу проекта;
- `/editor` — страницу редактора;
- `/preview` — страницу renderer preview.

Допускается также отдельная демонстрационная страница для media/gallery/navigation сценариев, если это упрощает тестирование сложных блоков.

## 37. Минимальный набор smoke-checks и финальная позиция по Nuxt

### 37.1. Минимальный набор ручных smoke-checks

После каждого заметного изменения в архитектуре или инфраструктуре желательно вручную проверять как минимум:

- старт dev-режима;
- build static/prerender версии;
- открытие editor-страницы;
- открытие preview-страницы;
- сохранение draft в localStorage;
- reset draft;
- export JSON;
- базовый renderer;
- публикацию на GitHub Pages.

### 37.2. Минимальный набор smoke-checks для editor UX

Для editor UX желательно регулярно проверять:

- создание блока;
- удаление блока;
- перемещение блока;
- комбинации inline tools;
- save/load;
- nested editor creation/destroy;
- media block interaction;
- sidebar navigation generation;
- отсутствие потери данных после пересоздания editor instance.

Когда появятся поздние слои, отдельно проверять:

- локализацию;
- тему;
- keyboard navigation;
- alt/caption в media scenarios;
- GitHub Pages after-deploy behavior.

### 37.3. Итоговая позиция по Nuxt-выбору

Использование Nuxt в текущем проекте считается оправданным при соблюдении следующих условий:

- проект остаётся статическим/prerender-ориентированным;
- текущий scope не размывается в сторону полноценной CMS;
- инфраструктурный слой Nuxt уменьшает будущую миграцию, а не усложняет editor-layer;
- GitHub Pages deployment остаётся рабочим и автоматизированным.

Текущий проект должен быть реализован как:

- **Nuxt static/prerender проект**;
- editor-focused sandbox с качественной архитектурой;
- промежуточный этап перед будущим **Nest + Nuxt** fullstack demo;
- проект, который автоматически публикуется на GitHub Pages и не требует ручной пересборки после каждого коммита.

Главная цель — не просто «запустить Editor.js», а получить:

- устойчивую editor architecture;
- понятную renderer architecture;
- воспроизводимый static deployment;
- задел для следующего проекта без лишней миграции.
